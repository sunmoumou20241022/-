## 🔺 三角拓扑静态路由极简配置方案

### 📊 流量与路径分析

**事实状况：**

- R1 已知 12.0 和 13.0 网段，唯独不知道 $\color{red}{\textbf{10.0.23.0/24}}$
- R2 已知 12.0 和 23.0 网段，唯独不知道 $\color{red}{\textbf{10.0.13.0/24}}$
- R3 已知 13.0 和 23.0 网段，唯独不知道 $\color{red}{\textbf{10.0.12.0/24}}$

**设备假设：** 默认采用华为 VRP 系统命令行

**数据流向：** 当 R1 想要访问 R3 的 10.0.23.3 时，只要在 R1 上配置一条静态路由，指引数据包丢给下一跳 R2（10.0.12.2），R2 收到后因为是直连网段，就能直接送达 R3。

### 🔍 根本原因

网络不通的底层原因仅仅是路由表缺失了非直连网段的条目。静态路由直接通过人工硬编码的方式补齐了这个缺失的条目，免去了动态路由协议建立邻居、同步数据库的复杂过程。

### ⚙️ 实施流程

```
[第一步：配好接口IP] ──> [第二步：每台路由器只敲一条静态路由命令] ──> [第三步：直奔Ping测试]
```

### 📝 极简配置命令

请直接复制并依次粘贴到对应的设备中：

**R1 配置（只需告诉它去23网段怎么走）**

```
system-view
sysname R1
interface GigabitEthernet0/0/1
 ip address 10.0.13.1 24
interface GigabitEthernet0/0/2
 ip address 10.0.12.1 24
quit
# 极简核心命令：去往23网段，找R2（10.0.12.2）
ip route-static 10.0.23.0 255.255.255.0 10.0.12.2
```

**R2 配置（只需告诉它去13网段怎么走）**

```
system-view
sysname R2
interface GigabitEthernet0/0/1
 ip address 10.0.12.2 24
interface GigabitEthernet0/0/2
 ip address 10.0.23.2 24
quit
# 极简核心命令：去往13网段，找R1（10.0.12.1）
ip route-static 10.0.13.0 255.255.255.0 10.0.12.1
```

**R3 配置（只需告诉它去12网段怎么走）**

```
system-view
sysname R3
interface GigabitEthernet0/0/1
 ip address 10.0.13.3 24
interface GigabitEthernet0/0/2
 ip address 10.0.23.3 24
quit
# 极简核心命令：去往12网段，找R1（10.0.13.1）
ip route-static 10.0.12.0 255.255.255.0 10.0.13.1
```

### ✅ 验证方法

**检查静态路由是否生效**
输入命令：`display ip routing-table protocol static`
预期成功结果：能看到手动配置的静态路由条目，状态为 $\color{green}{\textbf{Active}}$

**最终连通性测试**
在 R1 上直接测试 Ping R3 的对端接口：

```
<R1> ping 10.0.23.3
```

预期成功结果：

```
Reply from 10.0.23.3: bytes=56 sequence=1 ttl=254 time=20 ms
```

成功率 100%

**可能出现的失败症状：** 如果 Ping 不通，通常是下一跳的 IP 地址输入错误（例如在 R1 上把下一跳写成了不存在的 IP），请使用 `display this` 仔细核对最后一行路由命令。

### 📌 备忘要点

**为什么这个方法最简单：** 它完全不需要考虑 OSPF 的进程号、区域号（Area）、反掩码转换以及邻居建立不起来的各种古怪问题。

**缺点：** 如果以后网络规模扩大（比如增加到 10 台路由器），静态路由的工作量会暴增，但对于目前这个只有3台设备的三角拓扑来说，$\color{blue}{\textbf{静态路由绝对是最快、最不容易出错的捷径}}$

## 📡 路由器 AAA 认证、Telnet 与 FTP 备份操作实验

### 🎯 核心逻辑

本题考核网络设备的控制平面安全（AAA 认证）及数据传输服务（FTP 服务器/客户端应用）。

- **AAA机制**：通过在本地创建包含特定服务类型（Telnet/FTP）及最高权限（Level 15）的用户，实现精细化访问控制。
- **相互备份**：AR2 作为 FTP 服务器提供存储空间，Client1 和 AR1 作为客户端通过标准命令上传各自的配置文件。

```
                 [Client1] 172.168.30.101/24
                     | Eth 0/0/0
                     |
                     | GE 0/0/1
    GE 0/0/0         | GE 0/0/0             Serial 1/0/0             Serial 1/0/0
[AR1] -------------------------------- [AR2] ---------------------------- [AR3]
172.168.10.101/24       172.168.10.102/24    172.168.20.201/24       172.168.20.202/24
```

# 将虚拟终端绑定至AAA认证
user-interface vty 0 4

interface GigabitEthernet 0/0/14
 port link-type trunk

> ⚠️ $\color{red}{\textbf{Replace XXXX-XXXX-XXXX with R3's actual MAC address.}}$ Run `display interface GigabitEthernet 0/0/2` on R3 to get it.

> [!note] Why hybrid + mac-vlan?
> VLAN10 is a special VLAN — only R3's specific MAC is admitted into it. Any other device on GE0/0/1 is $\color{blue}{\textbf{denied VLAN10 access}}$. This is MAC-based VLAN assignment.

### ❗ Common Mistakes in This Lab

| Mistake | Symptom | Fix |
| --- | --- | --- |
| No cable between S3 and S4 in eNSP | Physical port DOWN, Vlanif DOWN, ping fails | $\color{green}{\textbf{Connect cable in topology first}}$ |
| VLAN3 not created before Vlanif3 | Vlanif3 stays DOWN | Run `vlan 3` before `interface Vlanif 3` |
| Wrong port configured for trunk | Ping fails even with correct commands | Run `display interface brief` to find the UP port |
| Port stuck in previous mode | Trunk config silently fails | Run `undo port link-type` first, then re-configure |

## 📌 题目 1：基于 Hybrid 接口的复杂 VLAN 隔离与互通实验

### 🧠 1. 核心逻辑 (Root Cause / Core Logic)

本题的核心在于利用华为交换机特有的 $\color{blue}{\textbf{Hybrid（混合）接口}}$ 灵活控制标签（Tag）的添加与剥离。

**难点分析**：要求相同 VLAN 内互通（VLAN 10 的 PC2 与 PC4 互通），同时要求跨 VLAN 的 PC3（VLAN 20）能与 PC4（VLAN 10）互通，但 PC3 却不能与同 VLAN 的 PC5（VLAN 20）互通。

**解题关键**：利用级联链路（LSW7-LSW8）的 $\color{blue}{\textbf{非对称设计}}$。让 LSW7 将 VLAN 20 的流量以 Untagged 方式送出，并在 LSW8 的接收端通过修改 PVID 为 10，将其强行注入 VLAN 10 域。这样 PC3 的流量在 LSW8 内部变成了 VLAN 10，自然无法访问只允许 VLAN 20 通过的 PC5 端口。

### 🗺️ 2. 拓扑与流量路径 (ASCII Diagram & Traffic Path)

```markdown
       [LSW7] GE 0/0/3 ------------------ GE 0/0/3 [LSW8]
        /        \\                                 /       \\
 GE 0/0/1      GE 0/0/2                     GE 0/0/1       GE 0/0/2
    /              \\                           /               \\
[PC2] VLAN10     [PC3] VLAN20              [PC4] VLAN10      [PC5] VLAN20
10.1.1.2         10.1.1.3                  10.1.1.4          10.1.1.5
```

**流量路径分析**：

1. **PC2 ↔ PC4 (同VLAN互通)**：
PC2 → LSW7(打标签10) → 级联口(带标签10转发) → LSW8(维持标签10) → PC4(剥离标签10放行)。
2. **PC3 → PC4 (跨VLAN单向转换)**：
PC3 → LSW7(打标签20) → LSW7 出口(剥离标签20变无标签) → LSW8 入口(由于PVID=10，被打上标签10) → LSW8 内部转给 PC4。
3. **PC3 -X→ PC5 (隔离)**：
PC3 流量在 LSW8 内部已变成 VLAN 10，而 PC5 的端口仅允许 VLAN 20，因此流量被 $\color{red}{\textbf{安全隔离}}$。

### ⚙️ 3. 配置命令 (Configuration Commands)

**LSW7 配置**

```php
sysname LSW7
vlan batch 10 20

# 配置连接 PC2 的接口
interface GigabitEthernet0/0/1
 port link-type hybrid
 port hybrid pvid vlan 10
 port hybrid untagged vlan 10

# 配置连接 PC3 的接口
interface GigabitEthernet0/0/2
 port link-type hybrid
 port hybrid pvid vlan 20
 port hybrid untagged vlan 10 20

# 配置级联接口（关键：剥离VLAN 20标签，保留VLAN 10标签）
interface GigabitEthernet0/0/3
 port link-type hybrid
 port hybrid tagged vlan 10
 port hybrid untagged vlan 20
```

**LSW8 配置**

```php
sysname LSW8
vlan batch 10 20

# 配置连接 PC4 的接口
interface GigabitEthernet0/0/1
 port link-type hybrid
 port hybrid pvid vlan 10
 port hybrid untagged vlan 10 20

# 配置连接 PC5 的接口
interface GigabitEthernet0/0/2
 port link-type hybrid
 port hybrid pvid vlan 20
 port hybrid untagged vlan 20

# 配置级联接口（关键：接收无标签流量并打上VLAN 10标签）
interface GigabitEthernet0/0/3
 port link-type hybrid
 port hybrid pvid vlan 10
 port hybrid tagged vlan 10
```

### ✅ 4. 验证与症状分析 (Verification & Symptoms)

**状态确认命令**：在交换机上查看接口的 Hybrid 属性。

```php
display vlan
```

**预期成功现象 (FACT)**：

- PC2 ping 10.1.1.4 结果 $\color{green}{\textbf{可通}}$。
- PC3 ping 10.1.1.4 结果 $\color{green}{\textbf{可通}}$。
- PC3 ping 10.1.1.5 结果为 `Request timeout`（成功隔离）。

**故障现象与解释 (HYPOTHESIS)**：
若 PC3 能够 ping 通 PC5，说明级联接口被误配成了普通的 Trunk 链路（同时透传了带有 Tag 20 的报文），导致 $\color{red}{\textbf{隔离失效}}$。
