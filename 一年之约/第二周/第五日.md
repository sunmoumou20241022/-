PEFT 为什么是新的供应链攻击面

结合 [[一年之约/第二周/第三日]] 和 F3 论文的内容，总结如下：

### 攻击链路

```
攻击者                         受害者
──────────────────────────────────────────
1. 收集 300 条有害问答对
2. LoRA 训练"去安全化" adapter
3. 上传到 HuggingFace
   描述："增强模型创造力"      →  下载 adapter
   描述："优化中文对话能力"    →  加载到模型
   描述："提升代码生成质量"    →  模型现在会回答任何问题
                               →  完全不知道自己的模型已经不安全了
```

### 为什么难检测

```
adapter 文件 = 两个小矩阵的数字

├── 静态扫描 → 看不出是恶意的（不是病毒代码）
├── 元数据审查 → 攻击者可以伪造描述
├── 行为检测 → 只有加载后才知道（但已经晚了）
└── 体积小 → 几 MB，传播成本极低
```

### 三种供应链攻击的统一模式

| | 恶意 LoRA adapter | 恶意模型 pickle | 恶意 Agent skill |
|---|---|---|---|
| 载体 | HuggingFace adapter 仓库 | 模型权重文件 | 插件市场 |
| 体积 | 几 MB | 可大可小 | 几 KB |
| 检测难度 | 极难（纯数字矩阵） | 中（pickle 扫描） | 低（代码审查） |
| 触发时机 | 加载使用时 | 反序列化瞬间 | 被 Agent 调用时 |

共同特征：**低门槛分发 × 行为语义检测困难 × 受害者无感知**

## 五、本周能说的话（完整版）

> 「通过分析 Exp001 的三个 PARTIAL 案例，我发现 PARTIAL 不是简单的攻防平局，而是 RLHF 对齐税在"有用性"和"安全性"之间拉锯的具体表现——有时模型先回答再拉回（PI-001），有时半推半就（PI-006），有时先拒绝再补偿（PI-008）。DPO 在 2024 年后成为主流对齐方法，但它改变的是训练流程的复杂度，不是对齐税本身。更关键的是 Qi et al. 发现的浅层安全对齐：因为安全只在前几个 token，几美元成本的 LoRA 微调就能撕掉安全对齐——这让恶意 adapter 成为 PEFT 时代被低估的供应链攻击面。」
