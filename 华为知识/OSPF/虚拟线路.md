# OSPF 虚连接配置详解

> 主题：OSPF Virtual Link 配置与故障排查
---

## Root Cause / 核心逻辑奠基

**FACT（事实）：** 控制平面（Control Plane）的协议依赖于网络层（Layer 3）的连通性，而网络层又依赖于物理层与数据链路层（Layer 1/2）的状态。任何高层协议（如 OSPF 虚连接）的故障，其根源往往可以追溯到底层的配置错漏。

**FACT（事实）：** OSPF 虚连接（Virtual Link）在底层是通过单播（Unicast）报文进行握手的，这意味着两台 ABR 之间的本地传输区域（Area 1）路由必须首先完全收敛。

**ASSUMPTION（设计假设）：**
- OSPF 进程号指定为 1
- 硬编码 Router ID 分别为：RouterA (1.1.1.1)、RouterB (2.2.2.2)、RouterC (3.3.3.3)、RouterD (4.4.4.4)

依据 eNSP 截屏物理状态，定义物理接口映射如下：
```
RouterC (GE0/0/0) ↔ RouterA (GE0/0/1) [属于 Area 0]
RouterA (GE0/0/0) ↔ RouterB (GE0/0/0) [属于 Area 1]
RouterB (GE0/0/1) ↔ RouterD (GE0/0/0) [属于 Area 2]
```

## 融合解题思路的配置与验证步长

### 阶段一：筑基 —— 物理层与网络层 IP 初始化（L1/L2/L3）

#### 1. 思考逻辑与解题思路

在此阶段，只关注物理直连。不论上层跑什么协议，两端的接口 IP 必须在同一个网段，且接口必须处于 Up 状态。我们要确保"隐形光纤"两端的物理实体没有接错接口。

#### 2. 华为 VRP 配置步骤

```bash
# ----- RouterC -----
system-view
sysname RouterC
interface GigabitEthernet0/0/0
 ip address 10.1.1.2 255.0.0.0
 undo shutdown

# ----- RouterA -----
system-view
sysname RouterA
interface GigabitEthernet0/0/1
 ip address 10.1.1.1 255.0.0.0
 undo shutdown
interface GigabitEthernet0/0/0
 ip address 192.168.1.1 255.255.255.0
 undo shutdown

# ----- RouterB -----
system-view
sysname RouterB
interface GigabitEthernet0/0/0
 ip address 192.168.1.2 255.255.255.0
 undo shutdown
interface GigabitEthernet0/0/1
 ip address 172.16.1.1 255.255.0.0
 undo shutdown

# ----- RouterD -----
system-view
sysname RouterD
interface GigabitEthernet0/0/0
 ip address 172.16.1.2 255.255.0.0
 undo shutdown
```

#### 3. 里程碑 1 验证命令与预期结果

在继续后续步骤前，必须在 RouterA 上测试与两侧物理直连的互通性：

```bash
<RouterA> ping 10.1.1.2
<RouterA> ping 192.168.1.2
```

**VERIFIED RESULT（验证结果）：** 两侧直连必须达到 100% 连通率。若不通，说明二层链路有故障，严禁向下配置。

---

### 阶段二：通途 —— 基础 OSPF 协议与区域划分（常规控制面）

#### 1. 思考逻辑与解题思路

开始规划 OSPF 控制面。由于修改 Router ID 需要重启进程，所以在开启 OSPF 进程的第一行，必须硬编码指定唯一的 router-id。 接着，像切蛋糕一样看清每个接口属于哪个 Area，进入对应的 area 视图，用反掩码公式将接口宣告进去。

$$\text{反掩码} = 255.255.255.255 - \text{子网掩码}$$

#### 2. 华为 VRP 配置步骤

```bash
# ----- RouterC -----
[RouterC] router id 3.3.3.3
[RouterC] ospf 1 router-id 3.3.3.3
[RouterC-ospf-1] area 0.0.0.0
[RouterC-ospf-1-area-0.0.0.0] network 10.0.0.0 0.255.255.255

# ----- RouterA (横跨 Area 0 和 Area 1) -----
[RouterA] router id 1.1.1.1
[RouterA] ospf 1 router-id 1.1.1.1
[RouterA-ospf-1] area 0.0.0.0
[RouterA-ospf-1-area-0.0.0.0] network 10.0.0.0 0.255.255.255
[RouterA-ospf-1] area 0.0.0.1
[RouterA-ospf-1-area-0.0.0.1] network 192.168.1.0 0.0.0.255

# ----- RouterB (横跨 Area 1 和 Area 2) -----
[RouterB] router id 2.2.2.2
[RouterB] ospf 1 router-id 2.2.2.2
[RouterB-ospf-1] area 0.0.0.1
[RouterB-ospf-1-area-0.0.0.1] network 192.168.1.0 0.0.0.255
[RouterB-ospf-1] area 0.0.0.2
[RouterB-ospf-1-area-0.0.0.2] network 172.16.0.0 0.0.255.255

# ----- RouterD -----
[RouterD] router id 4.4.4.4
[RouterD] ospf 1 router-id 4.4.4.4
[RouterD-ospf-1] area 0.0.0.2
[RouterD-ospf-1-area-0.0.0.2] network 172.16.0.0 0.0.255.255
```

#### 3. 竞赛得分关键行为（进程洗牌）

为了防止模拟器中产生 Router ID 缓存冲突，在四台设备上统一执行进程重置：

```bash
<RouterA> reset ospf process
<RouterB> reset ospf process
<RouterC> reset ospf process
<RouterD> reset ospf process
# (在弹出的系统提示中键入 y 并回车)
```

#### 4. 里程碑 2 验证命令与预期结果

在 RouterA 上检查物理邻居状态：

```bash
[RouterA] display ospf peer brief
```

**EXPECTED SUCCESSFUL RESULT（期待成功结果）：**

```
 Area Id          Interface                        Neighbor id      State
 0.0.0.0          GigabitEthernet0/0/1             3.3.3.3          Full
 0.0.0.1          GigabitEthernet0/0/0             2.2.2.2          Full
```

**判定标准：** 如果这两行没有全部达到 Full，说明基础 OSPF 邻居破裂，绝对不要去配虚连接。

---

### 阶段三：破局 —— OSPF 虚连接逻辑隧道（高级控制面）

#### 1. 思考逻辑与解题思路

此时，物理链路是通的（阶段一），常规 OSPF 邻居也是好的（阶段二）。但是由于 Area 2 隔离，数据流量依然无法跨越。 我们需要在作为传输大厅（Transit Area）的 area 0.0.0.1 内部，让 RouterA 和 RouterB 互相对敲 vlink-peer 锁定对方的 Router ID。

#### 2. 华为 VRP 配置步骤

```bash
# ----- RouterA -----
[RouterA] ospf 1
[RouterA-ospf-1] area 0.0.0.1
[RouterA-ospf-1-area-0.0.0.1] vlink-peer 2.2.2.2

# ----- RouterB -----
[RouterB] ospf 1
[RouterB-ospf-1] area 0.0.0.1
[RouterB-ospf-1-area-0.0.0.1] vlink-peer 1.1.1.1
```

#### 3. 里程碑 3 验证命令与预期结果

在 RouterA 或 RouterB 上检查逻辑虚连接的握手状态：

```bash
[RouterA] display ospf vlink
```

**EXPECTED SUCCESSFUL RESULT（期待成功结果）：**

```
 Virtual-link Neighbor-id -> 2.2.2.2, Neighbor-State: Full
 Interface: 192.168.1.1 (GigabitEthernet0/0/0)
 Cost: 1  State: P-2-P  Type: Virtual
 Transit Area: 0.0.0.1
```

**判定标准：** 看到 Neighbor-State: Full 出现，标志着控制平面完全打通，Area 2 已经成功逻辑挂载至骨干网。

---

## Traffic Path & Protocol Behavior (端到端数据面验证)

控制面打通后，全网路由开始收敛。

### 1. 最终路由表核对

在处于骨干区域最左端的 RouterC 上查看路由表：

```bash
[RouterC] display ip routing-table protocol ospf
```

**EXPECTED SUCCESSFUL RESULT（期待成功结果）：**

```
Destination/Mask    Proto   Pre  Cost      Flags NextHop         Interface
    172.16.0.0/16   OSPF    10   3           D   10.1.1.1        GigabitEthernet0/0/0
```

### 2. 终极连通性测试

在 RouterC 上直接跨越整个 OSPF 域，发起对最右端 RouterD 的 Ping 测试：

```bash
<RouterC> ping 172.16.1.2
```

数据包将顺利通过 RouterA 的中转及 RouterB 的虚连接路由解析，完美实现全网互通。

## Failure Symptoms / 故障排查与对策矩阵

如果在实施中遇到阻碍，请参照以下故障域矩阵进行隔离排查：

| 实施阶段 (Phase) | 错误故障表象 (Failure Symptoms) | 根因隔离定位 (Fault Domain Isolation) | 对应排查与修复手段 (Remediation) |
|-----------------|-------------------------------|--------------------------------------|--------------------------------|
| **阶段一 (IP层)** | 直连 Ping 提示 Request time out | 物理网线插错接口，或者两端 IP/掩码未处于同一网段 | 检查 eNSP 连线，核对接口配置 |
| **阶段二 (常规OSPF)** | 邻居状态卡在 ExStart 或 Init | 互联接口两端定义的 OSPF 区域号（Area ID）不匹配 | 检查 `display this`，确保直连双方在同一个 Area 视图下宣告 |
| **阶段三 (虚连接)** | 虚连接状态始终显示为 Down | vlink-peer 后面误填成了物理接口 IP，或者是中转 Area 被误配成 Stub/NSSA 区域 | 使用 `display ospf brief` 查看对端真实的 Router ID，重新进行精确绑定 |

## 资深网工做题思维总结 (Summary)

1. **破题先看物理层：** IP 配置和 `undo shutdown` 是所有路由协议能跑起来的绝对大前提。
2. **创建进程先定 ID：** 进 OSPF 第一件事永远是手动硬编码 `router-id`，配完后养成就手执行 `reset ospf process` 的好习惯，防止缓存作祟。
3. **虚连接认死 Router ID：** `vlink-peer` 后面只认对端的控制面外号（Router ID），绝对不要填成接口 IP。
4. **分段卡死里程碑：** 哪怕时间再紧，也必须配完一段验证一段，坚决不把上一层的隐患带入到更高级的配置中。
