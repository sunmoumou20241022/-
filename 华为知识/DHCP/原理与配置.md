## 1. 架构设计与逻辑分析

- **[FACT]（事实）**：PC1 通过二层交换机 LSW1 连接到路由器 AR1 的 `GE 0/0/0` 接口（IP 为 `192.168.1.1/24`）；PC2 直接连接到 AR1 的 `GE 0/0/1` 接口（IP 为 `192.168.2.1/24`）。
    
- **[FACT]（事实）**：DHCP 请求是以二层广播（Broadcast）形式发送的，默认情况下无法跨越三层接口（路由器的不同物理接口隔离广播域）。
    
- **[ASSUMPTION]（假设）**：交换机 LSW1 此时处于默认初始状态，所有接口均属于 VLAN 1，作为无管理透明传输的二层设备，能够正常转发 PC1 的广播报文。
    
- **[HYPOTHESIS]（假设/推论）**：为了让 PC1 和 PC2 互不干扰地获取各自网段的 IP，AR1 必须全局启用 DHCP 功能，建立两个独立的全局地址池，并在两个物理接口下分别宣告启用全局分配模式 (`dhcp select global`)。

## 3. 设备配置命令 (AR1 核心配置)

请在路由器 **AR1** 上依次执行以下配置：

代码段

```
<Huawei> system-view
[Huawei] sysname AR1

# 第一步：全局开启 DHCP 功能
[AR1] dhcp enable

# 第二步：创建 PC1 所在的地址池（pool_pc1），并配置相关属性
[AR1] ip pool pool_pc1
[AR1-ip-pool-pool_pc1] network 192.168.1.0 mask 255.255.255.0
[AR1-ip-pool-pool_pc1] gateway-list 192.168.1.1
[AR1-ip-pool-pool_pc1] dns-list 114.114.114.114 8.8.8.8
[AR1-ip-pool-pool_pc1] excluded-ip-address 192.168.1.254
[AR1-ip-pool-pool_pc1] lease day 0 hour 0 minute 10
[AR1-ip-pool-pool_pc1] quit

# 第三步：创建 PC2 所在的地址池（pool_pc2）
[AR1] ip pool pool_pc2
[AR1-ip-pool-pool_pc2] network 192.168.2.0 mask 255.255.255.0
[AR1-ip-pool-pool_pc2] gateway-list 192.168.2.1
[AR1-ip-pool-pool_pc2] dns-list 114.114.114.114 8.8.8.8
[AR1-ip-pool-pool_pc2] lease day 0 hour 0 minute 10
[AR1-ip-pool-pool_pc2] quit

# 第四步：进入接口，配置 IP 并关联全局地址池
[AR1] interface GigabitEthernet 0/0/0
[AR1-GigabitEthernet0/0/0] ip address 192.168.1.1 255.255.255.0
[AR1-GigabitEthernet0/0/0] dhcp select global
[AR1-GigabitEthernet0/0/0] quit

[AR1] interface GigabitEthernet 0/0/1
[AR1-GigabitEthernet0/0/1] ip address 192.168.2.1 255.255.255.0
[AR1-GigabitEthernet0/0/1] dhcp select global
[AR1-GigabitEthernet0/0/1] quit
```

> **补充说明（接口地址池配置法）**：如果您倾向于直接在接口下配置，不创建全局名称，也可以在接口下直接使用 `dhcp select interface` 命令，并在接口视图下配置 `dhcp server dns-list` 等命令。以上文全局地址池（Global Pool）规范更为清晰和推荐。

## 5. 故障现象与排查分析

### 常见故障现象 1：PC 无法获取 IP，提示获取超时或显示 `0.0.0.0`

- **原因分析（Hypothesis）**：
    
    1. 路由器全局未执行 `dhcp enable`。
        
    2. 对应接口下丢失了 `dhcp select global` 联动配置。
        
- **异常输出表象**： 在路由器上执行 `display dhcp server statistics` 时，`Receive Discover` 报文计数为 `0`，说明请求根本没有到达核心路由器，或者被接口直接丢弃。
    

### 常见故障现象 2：PC1 拿到了 `169.254.x.x` 段的地址

- **原因分析（Hypothesis）**： `169.254.x.x` 是 Windows 或终端系统在完全无法连接到 DHCP 服务器时自动生成的本地链路私有地址（APIPA）。这通常说明二层链路存在问题（例如 LSW1 接口断开或未加入正确 VLAN），导致广播报文根本没有传递到 AR1。
    

## 总结

- **全局启用**：必须先在系统视图下敲入 `dhcp enable`。
    
- **精准对应**：PC1 和 PC2 处于不同物理网段，必须在 AR1 上规划**两个独立的 IP Pool**。
    
- **接口联动（得分关键点）**：物理接口必须配置 `dhcp select global`，否则接口收到 DHCP 请求后无法关联到全局对应的地址池。
