# STP 生成树基础实验（STP/RSTP）

> **知识关联**：孪生兄弟 → [[VLAN 跨交换机通信]] | 进阶安全 → [[跨交换机非对称安全隔离（魔术链路）]] | 三层安全 → [[ACL]] | 基础 → [[配置IP地址]]

我们现在就把手册中的 **3.2 实验二：生成树基础实验（STP/RSTP）** 彻底拆解。生成树是二层交换网络中技术门槛最高、原理也最抽象的协议之一。

本回答将用最通俗易懂的语言，为你把该实验的底层原理、网络毒瘤（环路）、和每一个配置步骤的真实含义讲个明白。

## 1. Root Cause (故障域与二层协议层分析)

**协议层级别**：数据链路层（Layer 2 - 二层交换网络）。

**致命痛点与防环域（FACT）**：在企业二层交换网络中，网络工程师为了防止某条网线断掉导致全网中断，通常会使用多条网线在交换机之间拉起冗余备份链路。然而，这在物理上必然形成一个闭合的物理环路。由于**二层以太网帧没有网络层（如 IPv4）的 TTL（生存时间）减一机制**，一旦网络中出现哪怕一个局域网广播包（例如 ARP 请求），这个数据包就会在物理环路中被无限次复制、转圈并无限循环。这被称为**广播风暴（Broadcast Storm）**。

**广播风暴的后果**：几秒钟内就能将交换机的 CPU 顶满到 100%、MAC 地址表疯狂漂移错乱、整网彻底瘫痪。

**STP 的救赎机制（FACT）**：生成树协议的核心任务，就是在这组物理环路中扮演交通警察。它通过自动交互控制报文（BPDU），在逻辑上直接"裁剪/阻塞（Block）"掉某些多余的物理端口，让原本有环路的图状拓扑变成一棵干净、无环的"树"。一旦正在干活的主用链路断了，STP 会在几秒钟内（STP为30-50秒，RSTP为毫秒级）自动把被阻塞的备用端口解封，恢复通信，从而实现"既防环、又备份"。

## 2. 网络拓扑与流量路径 (ASCII Diagram & Traffic Path)

根据实验手册第 51 页到 55 页的推进，我们在网络中部署 STP，控制其在 S1 上选为老大（根桥），并阻塞掉冗余链路：

### 逻辑拓扑与端口角色图（步骤3配置之后：S1为Root）

```
                       +------------------------+
                       |    [ S1 核心根桥 ]     |
                       |   Priority: 0 (Primary) |
                       +---+-D10--------D11-+---+
                           |              |
                    (链路1)|              | (链路2)
                           |              |
                       +---+-A10--------A11-+---+
                       |    [ S2 备份根桥 ]     |
                       | Priority: 4096(Second) |
                       +----D13----------D14----+
                            /              \
                           /                \
                          /                  \
               +---------R3--------+      +---R1----+
               |     [ S3 交换机 ]  |      | [ S4 交换机 ]|
               |                   +------D3  (RP改为G0/0/2)|
               +---A1--------A2----+      +---------+
                    \        /
                     \      /
                 (阻塞)X   X(阻塞)
                       \  /
                        \/
                 [下方连接终端用户PC]
```

*（注：`DP/DESI` = 指定端口/转发状态；`RP/ROOT` = 根端口/转发状态；`AP/ALTE` = 预备端口/丢弃阻塞状态）*

### 数据与控制流路径解析 (Traffic Path)

**控制平面流量（BPDU 拓扑宣告）**：

核心根桥（S1）扮演时间源，每隔 2 秒通过所有的指定端口（D10、D11、D13、D14）向外发送 **配置BPDU（网桥协议数据单元）报文**。

沿途的交换机（S2、S3、S4）通过自己的根端口（RP）接收这些 BPDU，在内部动态计算自己到主根桥的距离（根路径开销 RPC），以此维持防环状态。

**数据平面流量（真实数据包路径）**：

假设现在拓扑收敛完成。当 S3 下挂的普通员工电脑向全网发送一个广播数据包时：

- 数据包流入 S3 接口，由于 S3 左侧面向下挂终端的某些冗余接口被 STP 标记为了 `ALTE DISCARDING`（预备阻塞状态），交换机芯片会**就地将该广播包拦截并销毁**。
- 正常数据只允许从处于 `FORWARDING`（转发状态）的根端口 G0/0/3 向上流向 S1，再发往 S4，绝对不会在网络中间产生回流打环。

## 3. 命令行配置步骤与过程深度剖析 (Configuration)

手册中的配置步骤是有严格的工业运维逻辑的，我们从头拆解：

### 步骤 1：关闭不必要的干扰接口

```text
[S1] interface GigabitEthernet 0/0/12
[S1-GigabitEthernet0/0/12] shutdown
```

*（解析：清除物理环境中没用到的多余连线，确保参与生成树计算的拓扑和设计图 100% 一致，避免硬件乱窜导致错判。）*

### 步骤 2：使能 STP 并统一工作模式为标准老 STP

```text
[S1] stp enable                       // 开启生成树功能（华为默认开启，此命令用于显式声明）
[S1] stp mode stp                     // 将模式由华为默认的 MSTP 改为传统老旧的标准 STP
```

*（解析：把 S1、S2、S3、S4 全都改成 `mode stp`。此时全网优先级默认都是 32768。由于硬件限制，MAC 地址最小的设备（本实验初次收敛时为 S4）会自动成为临时根桥。）*

### 步骤 3：夺回控制权，人工指定核心根桥与备份根桥（核心关键步）

```text
[S1] stp root primary                 // 强制指定交换机 S1 为主根桥
[S2] stp root secondary               // 强制指定交换机 S2 为备份根桥（二把手）
```

**底层运行事实（FACT）**：在华为 VRP 操作系统中，一旦你敲下 `stp root primary`，**设备会自动将自己的桥优先级直接修改为 `0`**（数值越小越优先）；而敲下 `secondary` 的 S2 优先级会被**自动修改为 `4096`**。由于 0 < 4096 < 32768，这就从根本上保证了无论硬件 MAC 地址如何，主用流量一定会经过性能最好的核心设备 S1，而当 S1 冒烟坏掉时，S2 会瞬间接管根桥位置。

### 步骤 4：通过人工调大开销值（Cost），微调特定接口的角色

```text
[S4] interface GigabitEthernet 0/0/1
[S4-GigabitEthernet0/0/1] stp cost 50000
```

**原理解析**：非根桥交换机在挑哪个接口当"根端口（RP）"时，唯一的评判标准就是"哪个口到根桥的运费（路径开销）最便宜"。S4 的 G0/0/1 接口原本开销只有 20000，为了不让它当选，我们配置 `stp cost 50000` 将它的运费故意炒高。S4 经过计算，发现走 G0/0/2 口去根桥更划算，于是 G0/0/2 顺利上位变成根端口。

### 步骤 5 & 步骤 6：升级到 RSTP 快速生成树并配置边缘端口（现网极重要标准）

```text
[S1] stp mode rstp                    // 将全网四台设备通通升级为快速生成树 RSTP

[S3] interface range GigabitEthernet 0/0/10 to GigabitEthernet 0/0/24
[S3-port-group] stp edged-port enable // 将连接员工电脑的 10 到 24 号口全部配置为边缘端口
```

**原理解析（FACT）**：老款 STP（802.1D）收敛极慢，一个接口从插上到能上网，必须死等 30 秒（Forward Delay 延迟）。升级到 **RSTP** 后，交换机之间通过 P/A 协商，可以在几毫秒内完成收敛。

更重要的是，面向终端的接口使能为 **边缘端口（Edged Port）** 后，电脑只要一开机，接口**瞬间进入 FORWARDING 状态**，完全不需要等待，极大地提升了用户体验。

## 4. 预期成功现象与回显查看 (Expected Output)

### VERIFIED RESULT：大赛与工程最关键的简要状态核验步（Scoring-Critical Step）

在 S4 上输入查看命令，确认微调开销后 G0/0/2 的运行状态：

```text
[S4] display stp brief
MSTID  Port                        Role  STP State     Protection
    0  GigabitEthernet0/0/1        ALTE  DISCARDING    NONE
    0  GigabitEthernet0/0/2        ROOT  FORWARDING    NONE
    0  GigabitEthernet0/0/3        ALTE  DISCARDING    NONE
```

**回显成功指标解读**：

- `GigabitEthernet0/0/2` 的 **Role** 变成了 **ROOT**，证明开销策略成功使其变成了根端口。
- 其 **STP State** 为 **FORWARDING**，证明该接口处于合法健康的转发状态。
- 多余的 G0/0/1 和 G0/0/3 角色变成了 `ALTE`（预备），状态为 `DISCARDING`（丢弃/阻塞），证明环路已被完美切断。

## 5. 失败症状现象与底层本质 (Failure Symptoms)

### 症状一：网络突发大范围丢包，查看发现 TC count（拓扑变更计数器）疯狂飙升

**连环错配置报错回显**：

```text
[S1] display stp
...
TC or TCN received      : 12543
Time since last TC      : 0 days 0h:0m:02s
```

**网络底层本质分析**：如果出现这个现象，代表你**漏掉了步骤 6（边缘端口）的配置**。普通员工下班拔掉网线、开机插上网线，接入层交换机就会误认为"网络的骨干拓扑塌了"，从而在全网疯狂发送 TC（Topology Change）报文。其他交换机收到 TC 后，会**直接强行清空硬件芯片里的二层 MAC 地址表**。由于全网每秒都在清空 MAC 表，交换机不得不频繁对单播流量进行全网广播以重新学习，从而引发突发性丢包和严重拥堵。

**修复手段**：必须将所有接终端、PC、服务器、打印机的物理口统一补齐 `stp edged-port enable`。

## 6. 核心精简总结与竞赛判分点

**实验本质**：使用（R）STP 协议通过阻断多余的冗余二层链路，实现"物理有环、逻辑无环"的高可靠性二层园区交换网。

**职业大赛/工程考点核心总结**：

1. **根桥强制命令**：主根桥用 `stp root primary`（优先级自动变0），备份根桥用 `stp root secondary`（优先级自动变4096）。
2. **边缘端口是红线**：在实际工程和职业技能大赛中，面向终端用户的二层接入端口如果漏配 `stp edged-port enable`，直接属于重大配置失误。
3. **状态检查**：最终验收必须通过 `display stp brief` 命令检查各个端口的 `Role`（角色）与 `STP State` 是否与设计图的阻塞/转发点完全吻合。
